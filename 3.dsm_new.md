## 들어가며

1부에서는 PlantUML로 **클래스 간 정적 의존 관계**를, 2부에서는 SequenceDiagram과 Call Hierarchy로 **메서드 단위의 동적 호출 흐름**을 시각화했습니다.

그런데 프로젝트 규모가 커지면 또 다른 차원의 질문이 등장합니다.

- *"order 패키지와 payment 패키지가 서로를 참조하고 있는 것 같은데, 정말 그런가?"*
- *"이 모듈을 떼어내서 별도 서비스로 분리할 수 있을까? 의존성이 얼마나 얽혀 있지?"*
- *"패키지 구조를 이렇게 나눈 게 맞긴 한 건가?"*

이런 질문에 답하려면 **패키지(모듈) 단위의 의존성**을 조감도처럼 한눈에 볼 수 있어야 합니다. 클래스 다이어그램은 클래스가 수십 개만 넘어가도 스파게티처럼 얽혀서 읽기 어렵고, 시퀀스 다이어그램은 하나의 호출 체인만 추적하니 전체 구조를 보여주지 못합니다.

여기서 등장하는 것이 **DSM(Dependency Structure Matrix)**입니다.

### DSM이란?

DSM은 소프트웨어 컴포넌트 간의 의존 관계를 **행렬(Matrix)** 형태로 표현하는 분석 기법입니다. 다이어그램과 달리 선이 꼬이지 않고, 수백 개의 패키지 관계도 하나의 표에 깔끔하게 담깁니다.

| 분석 도구 | 시각화 방식 | 적합한 규모 |
|-----------|-----------|------------|
| PlantUML 클래스 다이어그램 (1부) | 박스 + 화살표 | 클래스 5~20개 |
| SequenceDiagram (2부) | 시퀀스 화살표 | 단일 호출 체인 |
| **DSM (3부)** | **행렬 (표)** | **패키지 수십~수백 개** |

IntelliJ Ultimate에는 이 DSM 기능이 내장되어 있어서 별도 플러그인 설치 없이 바로 사용할 수 있습니다.

## 1. DSM 실행 방법

### 1.1 실행 경로

IntelliJ에서 DSM을 여는 방법은 여러 가지가 있습니다.

**방법 1: 메뉴에서 실행**

`Code → Analyze Code → Dependencies Matrix...` 를 선택합니다.

> 📸 *[스크린샷: Code 메뉴 → Analyze Code → Dependencies Matrix... 경로]*

**방법 2: Project 패널에서 실행**

좌측 Project 패널에서 분석하고 싶은 **패키지 또는 모듈을 우클릭** → `Analyze → Dependencies Matrix...` 를 선택합니다.

> 📸 *[스크린샷: Project 패널에서 패키지 우클릭 → Analyze → Dependencies Matrix...]*

**방법 3: Find Action으로 실행**

`Shift` 두 번 또는 `Ctrl+Shift+A` (macOS: `⌘+Shift+A`)로 Find Action을 열고, **"DSM"** 또는 **"Dependencies Matrix"**를 검색합니다.

> 📸 *[스크린샷: Find Action에서 "DSM" 검색 결과]*

### 1.2 분석 범위(Scope) 설정

실행하면 분석 범위를 묻는 대화상자가 나타납니다.

> 📸 *[스크린샷: Specify Analysis Scope 대화상자]*

| Scope 옵션 | 설명 | 추천 상황 |
|------------|------|----------|
| **Whole project** | 프로젝트 전체를 분석 | 전체 아키텍처 조감도가 필요할 때 |
| **Module 'xxx'** | 특정 모듈만 분석 | 멀티 모듈 프로젝트에서 하나의 모듈에 집중할 때 |
| **Directory 'xxx'** | 특정 디렉토리(패키지)만 분석 | 특정 도메인 패키지 내부를 분석할 때 |
| **Custom scope** | 직접 필터 조건 설정 | 테스트 코드 제외 등 세밀한 조정이 필요할 때 |

우리 예제에서는 `src/main/java` 하위의 전체 소스를 대상으로 분석합니다. 테스트 코드는 빠지는 편이 깔끔하니, `Production files` 또는 `Project Production Files` scope를 선택하는 것이 좋습니다.

### 1.3 DSM 화면 기본 구성

분석이 완료되면 아래와 같은 행렬 화면이 나타납니다.

> 📸 *[스크린샷: DSM 매트릭스 전체 화면 — 패키지 레벨]*

화면을 처음 보면 숫자가 빼곡한 표라서 당황할 수 있는데, 구조는 단순합니다.

```
                  order  payment  inventory  notification  common
order               -      3        2           1           5
payment             1      -        0           0           3
inventory           0      0        -           0           2
notification        0      0        0           -           2
common              0      0        0           0           -
```

읽는 법은 이렇습니다:

- **행(Row)**: 의존하는 쪽 (사용하는 쪽)
- **열(Column)**: 의존되는 쪽 (사용당하는 쪽)
- **셀의 숫자**: 의존 관계의 개수 (클래스/메서드 참조 횟수)

예를 들어, `order` 행 × `payment` 열에 `3`이 있으면 → **order 패키지가 payment 패키지를 3곳에서 참조**한다는 뜻입니다.

핵심 포인트: **대각선 아래(좌하단)에 숫자가 있으면 정상적인 하향 의존**, **대각선 위(우상단)에 숫자가 있으면 역방향 의존(= 순환 참조 후보)**입니다. IntelliJ는 이 역방향 의존을 **빨간색으로 강조** 표시해 줍니다.

> 📸 *[스크린샷: DSM에서 빨간색으로 표시된 역방향 의존 셀]*

### 1.4 셀 클릭으로 상세 확인

행렬의 숫자 셀을 클릭하면, 해당 의존 관계를 구성하는 **구체적인 클래스와 참조 목록**이 하단 패널에 표시됩니다.

> 📸 *[스크린샷: DSM 셀 클릭 시 하단에 나타나는 의존 관계 상세 목록]*

예를 들어, `order → payment` 셀(숫자 3)을 클릭하면:

```
OrderFacade → PaymentService.processPayment()
OrderFacade → PaymentResult
OrderService → PaymentStatus
```

이런 식으로 **어떤 클래스가 어떤 클래스를 참조하고 있는지** 구체적으로 보여줍니다. 이것이 DSM의 강력한 점 중 하나입니다. 숫자로 전체 조감도를 보다가, 궁금한 부분은 클릭 한 번으로 상세 내역까지 드릴다운할 수 있습니다.

### 1.5 패키지 펼치기/접기

DSM 왼쪽의 트리 구조에서 패키지를 **펼치면(expand)** 하위 클래스 단위까지 행렬을 확인할 수 있고, **접으면(collapse)** 패키지 단위로 요약됩니다.

> 📸 *[스크린샷: payment 패키지를 펼쳐서 하위 클래스별 의존 관계를 확인하는 모습]*

분석 흐름은 보통 이렇습니다: **패키지 단위**로 큰 그림을 먼저 파악 → 문제 있는 패키지를 **클래스 단위**로 펼쳐서 원인 추적.

## 3. DSM으로 발견하는 아키텍처 문제들

### 3.1 순환 참조 발견

DSM 매트릭스에서 가장 먼저 눈에 들어오는 것은 **빨간색 셀**입니다. 빨간색은 패키지 간 양방향 의존, 즉 **순환 참조(Circular Dependency)**를 의미합니다.

우리 프로젝트의 DSM을 보면 이런 형태가 나타납니다:

```
                  order  payment  inventory  notification  refund  common
order               -      3        2           1          0        5
payment            [1]     -        0           0          0        3
inventory           0      0        -           0          0        2
notification        0      0        0           -          0        2
refund              2     [1]       0           0          -        1
common              0      0        0           0          0        -
```

> **[1]로 표시된 셀이 빨간색(역방향 의존)입니다.**

두 가지 순환 참조가 발견됩니다:

**순환 참조 ①: order ↔ payment**

```
order → payment : OrderFacade가 PaymentService, PaymentResult 등을 참조 (3건)
payment → order : PaymentRetryService가 OrderService.findOrder()를 호출 (1건)
```

결제 재시도 로직에서 주문 정보를 조회하기 위해 `order` 패키지를 직접 참조하고 있습니다. 이러면 `payment` 패키지만 독립적으로 테스트하거나 분리하는 것이 불가능해집니다.

해당 셀을 클릭하면 구체적인 코드가 보입니다:

```java
// payment 패키지 안의 PaymentRetryService.java
@Service
@RequiredArgsConstructor
public class PaymentRetryService {

    private final PaymentRepository paymentRepository;
    private final PaymentService paymentService;
    private final OrderService orderService;  // ← 💥 order 패키지 직접 참조!

    public void reprocessPayment(String paymentId) {
        Payment payment = paymentRepository.findById(paymentId)
            .orElseThrow(() -> new PaymentNotFoundException(paymentId));

        // 주문이 아직 유효한지 확인하기 위해 OrderService를 직접 호출
        Order order = orderService.findOrder(payment.getOrderId());
        if (order.getStatus() == OrderStatus.CANCELLED) {
            payment.markAbandoned();
            paymentRepository.save(payment);
            return;
        }

        PaymentRequest retryRequest = PaymentRequest.from(order, payment.getMethod());
        paymentService.processPayment(retryRequest);
    }
}
```

**순환 참조 ②: refund ↔ payment**

```
refund → payment : RefundService가 PaymentService를 호출 (1건)
refund → order  : RefundFacade가 OrderService를 참조 (2건)
```

환불은 결제와 주문 모두에 의존하는데, 만약 `payment`도 `refund`를 참조하게 되면 삼각 순환이 생길 수 있습니다. 현재는 단방향이지만 주의가 필요한 구간입니다.

### 3.2 의존성 방향 분석 — "누가 누구를 더 많이 아는가"

DSM의 각 행(Row)의 합계를 보면 **해당 패키지가 얼마나 많은 외부 의존을 가지는지**를 알 수 있습니다.

```
패키지별 외부 의존 합계 (행 합계):
  order        : 3 + 2 + 1 + 0 + 5 = 11  ← 가장 많은 외부 의존
  payment      : 1 + 0 + 0 + 0 + 3 =  4
  refund       : 2 + 1 + 0 + 0 + 1 =  4
  inventory    : 0 + 0 + 0 + 0 + 2 =  2
  notification : 0 + 0 + 0 + 0 + 2 =  2
  common       : 0 + 0 + 0 + 0 + 0 =  0  ← 아무도 의존하지 않음 (이상적)
```

반대로 각 열(Column)의 합계는 **해당 패키지가 얼마나 많이 사용당하는지**를 보여줍니다.

```
패키지별 피의존 합계 (열 합계):
  common       : 5 + 3 + 2 + 2 + 1 = 13  ← 가장 많이 사용됨
  order        : 1 + 0 + 0 + 2 + 0 =  3
  payment      : 3 + 0 + 0 + 1 + 0 =  4
  inventory    : 2 + 0 + 0 + 0 + 0 =  2
  notification : 1 + 0 + 0 + 0 + 0 =  1
  refund       : 0 + 0 + 0 + 0 + 0 =  0  ← 아무도 사용하지 않음
```

이 숫자들이 알려주는 것:

- **`common`**: 열 합계가 13으로 가장 높고, 행 합계가 0입니다. 모두가 의존하지만 자신은 아무도 의존하지 않는 **유틸리티/기반 패키지**입니다. 이것은 이상적인 형태입니다.
- **`order`**: 행 합계가 11로 가장 높습니다. 다른 패키지를 가장 많이 알고 있는 **상위 조율 레이어**입니다. Facade 패턴을 쓰고 있으니 자연스러운 결과이지만, 이 숫자가 계속 커지면 God Package가 될 위험이 있습니다.
- **`refund`**: 열 합계가 0이라 아무도 이 패키지를 사용하지 않습니다. 독립적으로 분리하거나 제거해도 다른 패키지에 영향이 없다는 뜻입니다.

### 3.3 숨겨진 의존성 — common 패키지 비대화

DSM에서 `common` 열의 숫자를 보면 **모든 패키지가 common에 의존**하고 있습니다. 이것 자체는 문제가 아닙니다. 그런데 `common` 패키지를 펼쳐 보면 문제가 드러납니다.

> 📸 *[스크린샷: DSM에서 common 패키지를 펼친 모습]*

```
common 패키지 하위:
  common/
  ├── Money.java                          → order, payment, refund에서 사용
  ├── BaseEntity.java                     → 모든 도메인에서 사용
  └── exception/
      ├── OrderProcessingException.java   → order에서만 사용
      ├── PaymentFailedException.java     → order, payment에서 사용
      ├── InsufficientStockException.java → inventory에서만 사용
      ├── ProductNotFoundException.java   → inventory에서만 사용
      └── ReservationNotFoundException.java → inventory에서만 사용
```

`common`에 있는 예외 클래스들을 잘 보면, **특정 도메인에서만 사용되는 예외가 common에 들어가 있습니다.** `InsufficientStockException`은 inventory에서만 쓰는데 왜 common에 있을까요?

이런 패턴은 프로젝트 초기에 "예외는 공통이니까 한곳에 모아두자"라는 결정에서 시작되는 경우가 많습니다. 하지만 시간이 지나면 common 패키지가 **모든 도메인의 잡동사니 서랍**이 되어버립니다.

### 3.4 클래스 레벨로 드릴다운

패키지 레벨에서 문제를 발견했으면, 해당 패키지를 **펼쳐서 클래스 단위로 상세 분석**합니다.

`payment` 패키지를 펼쳐서 어떤 클래스가 `order`에 대한 역방향 의존을 만들고 있는지 확인합니다:

> 📸 *[스크린샷: payment 패키지를 펼친 DSM — PaymentRetryService 행에 빨간 셀]*

```
                           OrderService  OrderStatus  ...
PaymentService                0             0
PaymentValidator              0             0
PaymentGatewayRouter          0             0
PaymentRepository             0             0
PaymentRetryService          [1]           [1]         ← 💥 이 클래스가 범인!
PaymentRetryScheduler         0             0
```

`PaymentRetryService` 하나가 순환 참조의 원인입니다. 이처럼 DSM은 **패키지 레벨에서 문제를 발견하고, 클래스 레벨에서 원인을 특정**하는 흐름을 자연스럽게 지원합니다.

## 4. DSM 분석 결과를 리팩토링에 적용하기

DSM에서 발견한 문제별로 리팩토링 방향을 정리해 보겠습니다.

### 4.1 순환 참조 해소: order ↔ payment

**문제**: `PaymentRetryService`가 `OrderService`를 직접 참조하여 순환 참조 발생

**AS-IS**:

```java
// payment 패키지
@Service
@RequiredArgsConstructor
public class PaymentRetryService {

    private final PaymentRepository paymentRepository;
    private final PaymentService paymentService;
    private final OrderService orderService;  // ← order 패키지 직접 참조

    public void reprocessPayment(String paymentId) {
        Payment payment = paymentRepository.findById(paymentId)
            .orElseThrow(() -> new PaymentNotFoundException(paymentId));

        Order order = orderService.findOrder(payment.getOrderId());
        if (order.getStatus() == OrderStatus.CANCELLED) {
            payment.markAbandoned();
            paymentRepository.save(payment);
            return;
        }

        PaymentRequest retryRequest = PaymentRequest.from(order, payment.getMethod());
        paymentService.processPayment(retryRequest);
    }
}
```

**해결 전략: 인터페이스 역전(DIP)**

`payment` 패키지 안에 인터페이스를 두고, `order` 패키지에서 이를 구현하게 합니다.

```java
// payment 패키지에 인터페이스 정의
public interface OrderStatusQueryPort {
    boolean isOrderCancelled(String orderId);
}
```

```java
// order 패키지에서 구현
@Component
@RequiredArgsConstructor
public class OrderStatusQueryAdapter implements OrderStatusQueryPort {

    private final OrderRepository orderRepository;

    @Override
    public boolean isOrderCancelled(String orderId) {
        return orderRepository.findById(orderId)
            .map(order -> order.getStatus() == OrderStatus.CANCELLED)
            .orElse(true);
    }
}
```

```java
// payment 패키지 — 리팩토링 후
@Service
@RequiredArgsConstructor
public class PaymentRetryService {

    private final PaymentRepository paymentRepository;
    private final PaymentService paymentService;
    private final OrderStatusQueryPort orderStatusQueryPort;  // ← 인터페이스에만 의존!

    public void reprocessPayment(String paymentId) {
        Payment payment = paymentRepository.findById(paymentId)
            .orElseThrow(() -> new PaymentNotFoundException(paymentId));

        if (orderStatusQueryPort.isOrderCancelled(payment.getOrderId())) {
            payment.markAbandoned();
            paymentRepository.save(payment);
            return;
        }

        paymentService.processPayment(payment.toRetryRequest());
    }
}
```

**TO-BE DSM**:

```
                  order  payment
order               -      3     (변화 없음)
payment             0      -     ← 빨간 셀 사라짐!
```

`payment → order` 방향의 의존이 사라지고, `order → payment` 단방향만 남습니다. 인터페이스 `OrderStatusQueryPort`는 `payment` 패키지에 있지만 구현체는 `order`에 있으므로, **의존성 방향이 역전**됩니다.

> 📸 *[스크린샷: 리팩토링 후 DSM에서 빨간 셀이 사라진 모습 — Before/After 비교]*

### 4.2 common 패키지 정리: 도메인 전용 예외 이동

**문제**: 특정 도메인에서만 사용하는 예외가 common에 몰려 있음

**방향**: 각 예외를 사용하는 도메인 패키지로 이동

```
AS-IS:
common/exception/
├── OrderProcessingException.java      → order 패키지로 이동
├── PaymentFailedException.java        → payment 패키지로 이동 (order에서도 catch하지만, 발생원은 payment)
├── InsufficientStockException.java    → inventory 패키지로 이동
├── ProductNotFoundException.java      → inventory 패키지로 이동
└── ReservationNotFoundException.java  → inventory 패키지로 이동

TO-BE:
common/exception/
└── (비어 있음 — 또는 진짜 공통 예외만 남김)

common/
├── Money.java        ← 진짜 공통
└── BaseEntity.java   ← 진짜 공통
```

이 리팩토링 후 DSM의 `common` 열 숫자가 줄어들고, 각 도메인 패키지의 **자기 완결성**이 높아집니다.

단, `PaymentFailedException`처럼 여러 패키지에서 참조하는 예외는 신중하게 판단해야 합니다. `order` 패키지의 `OrderFacade`에서 이 예외를 catch하고 있기 때문에, `payment`으로 옮기면 `order → payment` 의존이 하나 늘어납니다. 이미 `order → payment` 의존이 존재하므로 새로운 의존 방향이 추가되는 것은 아니지만, 의존의 **밀도**가 높아지는 점은 인지해야 합니다.

### 4.3 모듈 분리 가능성 평가

DSM은 **"이 패키지를 별도 모듈/서비스로 분리할 수 있을까?"**를 판단하는 데에도 유용합니다.

`inventory` 패키지를 예로 들어보겠습니다:

```
inventory의 DSM 행(의존하는 곳):
  inventory → common : 2

inventory의 DSM 열(의존당하는 곳):
  order → inventory : 2
```

`inventory`는 `common`에만 의존하고, `order`만이 이를 사용합니다. 의존 관계가 매우 깔끔합니다. 이 패키지는 **별도 모듈로 분리하기 가장 좋은 후보**입니다.

반면 `order` 패키지는:

```
order의 DSM 행(의존하는 곳):
  order → payment      : 3
  order → inventory    : 2
  order → notification : 1
  order → common       : 5
```

4개 패키지에 의존하고 있어서, 단독 분리 시 인터페이스를 많이 정의해야 합니다. 분리가 불가능한 건 아니지만, `inventory`보다 비용이 훨씬 높습니다.

이런 식으로 **행/열 합계와 의존 방향**을 보면 분리 가능성과 비용을 정량적으로 판단할 수 있습니다.

## 5. DSM 활용 팁

**💡 Tip 1: 대각선 기준으로 빨간색부터 확인**

DSM을 열자마자 빨간색 셀부터 찾으세요. 순환 참조는 거의 항상 리팩토링 1순위입니다.

**💡 Tip 2: 점진적으로 펼치기**

최상위 패키지 레벨에서 시작 → 문제 있는 패키지를 펼쳐서 클래스 단위 분석 → 셀 클릭으로 구체적 참조 확인. 한 번에 모든 클래스를 펼치면 행렬이 너무 커져서 의미를 파악하기 어렵습니다.

**💡 Tip 3: 리팩토링 전후 비교에 활용**

리팩토링 전에 DSM 스크린샷을 저장해 두고, 리팩토링 후 다시 생성해서 비교하면 **정량적 개선 효과**를 확인할 수 있습니다. 코드 리뷰나 팀 회의에서 리팩토링의 필요성을 설득할 때 특히 유용합니다.

**💡 Tip 4: 정기적으로 실행하기**

분기에 한 번 정도 DSM을 돌려보면, 자연스럽게 쌓이는 순환 참조나 의존성 비대화를 조기에 발견할 수 있습니다. "기술 부채 점검"의 일환으로 팀 루틴에 넣어도 좋습니다.

**💡 Tip 5: 패키지 정렬에 주의**

DSM 매트릭스에서 패키지의 순서를 **의존 방향에 따라 정렬**(상위 레이어가 위, 하위 레이어가 아래)하면 읽기가 훨씬 쉬워집니다. IntelliJ가 기본으로 알파벳 순 정렬을 하는데, 왼쪽 트리를 드래그해서 순서를 조정할 수 있습니다. 이상적인 의존 구조에서는 **모든 숫자가 대각선 아래에만** 위치합니다.

## 6. 1부~3부 도구 정리

3부작을 통해 리팩토링을 돕는 세 가지 시각화 도구를 살펴봤습니다. 각 도구가 답하는 질문과 적합한 분석 레벨을 정리합니다.

| | 1부: PlantUML | 2부: SequenceDiagram + Call Hierarchy | 3부: DSM |
|---|---|---|---|
| **분석 레벨** | 클래스/패키지 | 메서드/로직 | 아키텍처/모듈 |
| **시각화 형태** | 클래스 다이어그램 | 시퀀스 다이어그램 + 트리 | 의존성 행렬 |
| **핵심 질문** | "누가 누구를 알고 있나?" | "어떤 순서로 실행되나?" / "고치면 어디까지 영향?" | "패키지 간 순환 참조가 있나?" / "분리할 수 있나?" |
| **적합한 규모** | 클래스 5~20개 | 단일 호출 체인 | 패키지 수십~수백 개 |
| **리팩토링 활용** | 클래스 의존 관계 정리 | 메서드 시그니처 변경의 영향 범위 파악 | 순환 참조 해소, 모듈 분리 |

이 세 도구를 상황에 맞게 조합하면, **코드를 읽기 전에 구조를 이해하고, 코드를 고치기 전에 영향 범위를 파악하고, 코드를 고친 후에 개선 효과를 검증**하는 전체 사이클을 커버할 수 있습니다.

## 마치며

3부작에 걸쳐 리팩토링을 도와주는 IntelliJ 기반 도구들을 살펴봤습니다.

사실 이 도구들은 하나같이 **"새로운 기능"이 아닙니다.** PlantUML도, SequenceDiagram 플러그인도, DSM도 이미 오래전부터 존재하던 도구들이죠. 그런데 실제로 리팩토링 현장에서 이것들을 적극적으로 활용하는 개발자는 생각보다 많지 않은 것 같습니다.

리팩토링은 "코드를 잘 고치는 것"이기도 하지만, 그 전에 **"코드를 정확히 이해하는 것"**에서 시작됩니다. 머릿속으로만 구조를 파악하려 하면 한계가 있고, 실수가 생깁니다. 도구가 보여주는 시각적 피드백은 그 한계를 상당히 줄여 줍니다.
