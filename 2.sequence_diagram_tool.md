## 들어가며

1부에서는 PlantUML을 활용해 **클래스 간 정적인 의존 관계**를 다이어그램으로 시각화하는 방법을 살펴봤습니다. 클래스가 "누구를 알고 있는가"를 파악하는 데는 큰 도움이 됐죠.

그런데 실제로 리팩토링을 하려고 코드에 손을 대면, 다른 종류의 질문이 떠오릅니다.

- *"이 메서드를 고치면 어디까지 영향이 갈까?"*
- *"`placeOrder()`를 호출하면 내부적으로 어떤 순서로 뭐가 실행되지?"*
- *"이 로직, 중간에 왜 갑자기 저쪽 서비스를 호출하는 거야?"*

이런 질문들은 **정적인 구조도만으로는 답하기 어렵습니다.** 클래스 다이어그램은 `OrderService`가 `PaymentService`를 의존한다는 "관계"는 보여주지만, 실행 시점에 **어떤 메서드가 어떤 순서로 호출되는지**는 알려주지 않거든요.

이번 2부에서는 바로 이 **"동적인 로직 흐름"**을 파악하는 두 가지 도구를 다룹니다.

| 도구 | 특징 | 활용 상황 |
|------|------|-----------|
| **SequenceDiagram 플러그인** | 메서드 호출 흐름을 시퀀스 다이어그램으로 자동 생성 | "이 메서드 내부에서 무슨 일이 일어나지?" |
| **Call Hierarchy** (IntelliJ 내장) | 특정 메서드를 누가 호출하는지 / 이 메서드가 뭘 호출하는지 트리로 탐색 | "이 메서드를 고치면 어디까지 영향이 갈까?" |

1부에서 사용했던 전자상거래 주문 처리 시스템 코드를 그대로 이어서 사용할 거예요. 이미 익숙한 코드니까 도구 자체에 집중할 수 있겠죠?

## 예제 코드 다시 보기

1부에서 만든 코드베이스 중, 오늘 집중할 클래스들을 다시 한번 살펴보겠습니다. 2부에서는 **메서드 내부 흐름**이 중요하니까 메서드 구현체에 초점을 맞춰볼게요.

### OrderFacade — 주문의 시작점

```java
@Component
@RequiredArgsConstructor
public class OrderFacade {

    private final OrderService orderService;
    private final PaymentService paymentService;
    private final InventoryService inventoryService;
    private final NotificationService notificationService;
    private final OrderEventPublisher orderEventPublisher;

    @Transactional
    public OrderResult placeOrder(OrderCommand command) {
        // 1) 재고 선점
        InventoryReservation reservation = inventoryService.reserve(
            command.getItems().stream()
                .map(item -> new ReservationRequest(item.getProductId(), item.getQuantity()))
                .toList()
        );

        try {
            // 2) 주문 생성
            Order order = orderService.createOrder(command, reservation);

            // 3) 결제 처리
            PaymentResult paymentResult = paymentService.processPayment(
                new PaymentRequest(
                    order.getId(),
                    order.getTotalAmount(),
                    command.getPaymentMethod()
                )
            );

            // 4) 주문 확정
            order = orderService.confirmOrder(order.getId(), paymentResult);

            // 5) 알림 발송
            notificationService.sendOrderConfirmation(order);

            // 6) 이벤트 발행
            orderEventPublisher.publish(new OrderPlacedEvent(order));

            return OrderResult.success(order);

        } catch (PaymentFailedException e) {
            inventoryService.releaseReservation(reservation.getId());
            orderService.markOrderFailed(command.getOrderId(), e.getReason());
            return OrderResult.failure(e.getReason());

        } catch (Exception e) {
            inventoryService.releaseReservation(reservation.getId());
            throw new OrderProcessingException("주문 처리 중 예기치 않은 오류 발생", e);
        }
    }
}
```

### PaymentService — 결제 처리

```java
@Service
@RequiredArgsConstructor
public class PaymentService {

    private final PaymentGatewayRouter gatewayRouter;
    private final PaymentValidator paymentValidator;
    private final PaymentRepository paymentRepository;
    private final RetryTemplate retryTemplate;

    public PaymentResult processPayment(PaymentRequest request) {
        // 유효성 검증
        paymentValidator.validate(request);

        // 결제 수단에 맞는 게이트웨이 선택
        PaymentGateway gateway = gatewayRouter.route(request.getPaymentMethod());

        // 재시도 포함 결제 실행
        PaymentResponse response = retryTemplate.execute(context ->
            gateway.charge(request.getAmount(), request.buildChargeDetails())
        );

        // 결제 이력 저장
        Payment payment = Payment.builder()
            .orderId(request.getOrderId())
            .amount(request.getAmount())
            .method(request.getPaymentMethod())
            .gatewayTransactionId(response.getTransactionId())
            .status(response.isSuccess() ? PaymentStatus.COMPLETED : PaymentStatus.FAILED)
            .build();

        paymentRepository.save(payment);

        if (!response.isSuccess()) {
            throw new PaymentFailedException(response.getFailureReason());
        }

        return PaymentResult.of(payment);
    }
}
```

### InventoryService — 재고 관리

```java
@Service
@RequiredArgsConstructor
public class InventoryService {

    private final InventoryRepository inventoryRepository;
    private final ReservationRepository reservationRepository;
    private final InventoryEventPublisher eventPublisher;

    @Transactional
    public InventoryReservation reserve(List<ReservationRequest> requests) {
        List<InventoryLock> locks = new ArrayList<>();

        for (ReservationRequest request : requests) {
            Inventory inventory = inventoryRepository
                .findByProductIdWithLock(request.getProductId())
                .orElseThrow(() -> new ProductNotFoundException(request.getProductId()));

            if (inventory.getAvailableQuantity() < request.getQuantity()) {
                // 이미 잡은 lock 해제
                locks.forEach(InventoryLock::release);
                throw new InsufficientStockException(request.getProductId(), 
                    inventory.getAvailableQuantity(), request.getQuantity());
            }

            InventoryLock lock = inventory.lock(request.getQuantity());
            locks.add(lock);
        }

        InventoryReservation reservation = InventoryReservation.create(locks);
        reservationRepository.save(reservation);

        eventPublisher.publish(new InventoryReservedEvent(reservation));
        return reservation;
    }

    @Transactional
    public void releaseReservation(String reservationId) {
        InventoryReservation reservation = reservationRepository
            .findById(reservationId)
            .orElseThrow(() -> new ReservationNotFoundException(reservationId));

        reservation.release();
        reservationRepository.save(reservation);

        eventPublisher.publish(new InventoryReleasedEvent(reservation));
    }
}
```

### NotificationService — 알림 발송

```java
@Service
@RequiredArgsConstructor
public class NotificationService {

    private final NotificationTemplateEngine templateEngine;
    private final NotificationChannelResolver channelResolver;
    private final NotificationRepository notificationRepository;

    @Async
    public void sendOrderConfirmation(Order order) {
        NotificationTemplate template = templateEngine.resolve("ORDER_CONFIRMATION", order);
        
        List<NotificationChannel> channels = channelResolver.resolveChannels(
            order.getCustomer(), NotificationType.ORDER_CONFIRMATION
        );

        for (NotificationChannel channel : channels) {
            try {
                NotificationMessage message = template.render(channel.getFormat());
                channel.send(message);
                
                notificationRepository.save(
                    NotificationRecord.success(order.getId(), channel.getType(), message)
                );
            } catch (NotificationDeliveryException e) {
                notificationRepository.save(
                    NotificationRecord.failure(order.getId(), channel.getType(), e.getMessage())
                );
            }
        }
    }
}
```

코드를 다시 훑어보니, `placeOrder()` 하나를 호출하면 **재고 → 주문 생성 → 결제 → 확정 → 알림 → 이벤트 발행**이라는 꽤 긴 체인이 돌아간다는 걸 알 수 있습니다. 이걸 머릿속으로만 따라가기엔 좀 벅차죠.

이제 이 흐름을 도구로 시각화해 보겠습니다.

## 1. SequenceDiagram 플러그인

### 1.1 이 도구가 왜 필요한가

시퀀스 다이어그램은 **객체 간 메서드 호출 순서**를 시간 흐름에 따라 보여주는 다이어그램입니다. 클래스 다이어그램이 "구조"를 보여준다면, 시퀀스 다이어그램은 **"행동"**을 보여줍니다.

리팩토링 관점에서 시퀀스 다이어그램이 유용한 이유는 명확합니다:

- **숨겨진 호출 체인 발견**: 코드를 읽으면서 하나하나 따라가지 않아도, 한눈에 전체 흐름이 보입니다.
- **과도한 책임 식별**: 하나의 메서드에서 너무 많은 외부 호출이 뻗어나가면 시각적으로 바로 티가 납니다.
- **리팩토링 전후 비교**: "이 단계를 분리하면 흐름이 어떻게 바뀔까"를 시각적으로 비교할 수 있습니다.

### 1.2 설치

IntelliJ에서 `Settings → Plugins → Marketplace`에서 **"SequenceDiagram"**을 검색하여 설치합니다.

> 📸 *[스크린샷: Marketplace에서 SequenceDiagram 검색 화면]*
> - 제작자: **Vanco** (https://github.com/nicedoc/SequencePlugin)
> - 설치 후 IDE 재시작

⚠️ **Free 버전 제한 사항**: 본 글에서는 무료 버전만 사용합니다. 무료 버전에서는 다이어그램의 **호출 깊이(call depth)**, **외부 클래스 필터링**, **PlantUML/Mermaid 소스 내보내기** 등 일부 고급 기능이 제한됩니다. 하지만 기본적인 시퀀스 다이어그램 생성과 확인에는 충분합니다.

### 1.3 기본 사용법

사용법은 아주 간단합니다.

**Step 1.** 시퀀스 다이어그램을 보고 싶은 **메서드 이름 위에 커서**를 놓습니다.

**Step 2.** 마우스 우클릭 → **`Sequence Diagram`** 선택 (또는 메서드 이름에 커서를 둔 상태에서 단축키 사용)

> 📸 *[스크린샷: placeOrder 메서드 위에서 우클릭 → Sequence Diagram 메뉴 선택]*

**Step 3.** 옵션 대화상자가 나타납니다.

> 📸 *[스크린샷: SequenceDiagram 옵션 대화상자]*

주요 옵션은 다음과 같습니다:

- **Call depth**: 몇 단계까지 추적할지 설정합니다. 기본값은 보통 1~2인데, 깊이를 늘리면 더 세밀한 흐름을 볼 수 있습니다.
  - `1`: `placeOrder`가 직접 호출하는 메서드만 표시
  - `2`: 그 메서드들이 내부에서 호출하는 메서드까지 표시
  - 너무 깊게 설정하면 다이어그램이 복잡해지니 **2~3 정도**를 추천합니다.
- **Project class only**: 체크하면 JDK/라이브러리 클래스 호출은 제외하고 **프로젝트 내 클래스만** 표시합니다. 리팩토링 분석에서는 거의 항상 체크하는 게 좋습니다.

**Step 4.** OK를 누르면 시퀀스 다이어그램이 생성됩니다.

### 1.4 placeOrder() 시퀀스 다이어그램 해석

`OrderFacade.placeOrder()`에 대해 **Call depth 2, Project class only** 옵션으로 생성하면 아래와 같은 형태의 다이어그램이 나타납니다.

> 📸 *[스크린샷: placeOrder() 시퀀스 다이어그램 전체 모습]*

다이어그램에서 확인할 수 있는 내용을 정리해 보면:

```

    |                               |                          |                     |                       |                      |
    |--- reserve(requests) -------->|                          |                     |                       |                      |
    |                               |-- findByProductId..() -->|[Repository]         |                       |                      |
    |                               |-- lock(quantity) ------->|[Inventory]          |                       |                      |
    |                               |-- save(reservation) ---->|[Repository]         |                       |                      |
    |<-- reservation ---------------|                          |                     |                       |                      |
    |                               |                          |                     |                       |                      |
    |--- createOrder(cmd, res) ------------------------------ >|                     |                       |                      |
    |<-- order ------------------------------------------------|                     |                       |                      |
    |                               |                          |                     |                       |                      |
    |--- processPayment(request) -------------------------------------------------->|                       |                      |
    |                               |                          |                     |-- validate() -------->|[Validator]           |
    |                               |                          |                     |-- route() ----------->|[Router]              |
    |                               |                          |                     |-- charge() ---------->|[Gateway]             |
    |                               |                          |                     |-- save(payment) ----->|[Repository]          |
```

텍스트로는 한계가 있지만, 실제 플러그인 화면에서는 이런 흐름이 **색상 구분된 화살표와 박스**로 깔끔하게 표현됩니다.

### 1.5 다이어그램에서 보이는 리팩토링 포인트

시퀀스 다이어그램을 한눈에 보면, 코드를 한 줄 한 줄 읽을 때는 쉽게 놓칠 수 있는 패턴이 드러납니다.

#### 🔍 포인트 1: OrderFacade의 과도한 오케스트레이션

다이어그램을 보면 `OrderFacade`에서 **6개의 서로 다른 객체로** 화살표가 뻗어나갑니다. 이건 이 메서드가 너무 많은 것을 직접 조율하고 있다는 시각적 신호입니다. Facade라는 이름에 걸맞게 단순화하거나, 일부 단계를 묶어서 별도 서비스로 분리하는 것을 고려해 볼 수 있습니다.

예를 들어:
- `결제 처리 + 주문 확정`을 하나의 트랜잭션 단위로 묶기
- `알림 발송 + 이벤트 발행`을 후처리(post-processing) 단위로 분리

#### 🔍 포인트 2: PaymentService 내부의 숨겨진 복잡도

Call depth를 2로 설정하면 `processPayment()` 내부에서 **4단계의 연쇄 호출**이 벌어지는 것이 보입니다:
`validate → route → charge → save`

이 중 `route`와 `charge`는 결제 수단에 따라 다른 게이트웨이로 분기되는 전략 패턴인데, 다이어그램에서 이 사실이 명확하게 드러납니다.

#### 🔍 포인트 3: 예외 발생 시 흐름

무료 버전에서는 try-catch 흐름까지 완벽하게 표현해 주지는 않지만, `placeOrder()` 코드를 보면 결제 실패 시 `releaseReservation()`을 호출하는 보상 트랜잭션이 있습니다. 이것은 나중에 Saga 패턴이나 별도의 롤백 전략으로 분리할 수 있는 후보입니다.

### 1.6 활용 팁

**💡 Tip 1: 진입점 메서드부터 시작하기**

Controller나 Facade의 최상위 메서드에서 시작하면 전체 흐름을 빠르게 파악할 수 있습니다.

**💡 Tip 2: Call depth 점진적으로 늘리기**

- depth 1로 먼저 "큰 그림"을 파악
- 관심 있는 부분을 발견하면 해당 메서드로 이동해서 depth 2~3으로 상세 확인

예를 들어, `placeOrder()`를 depth 1로 먼저 보고, `processPayment()`가 복잡해 보이면 `processPayment()`에 직접 커서를 올려 depth 2로 다시 생성하면 됩니다.

**💡 Tip 3: "Project class only" 적극 활용**

Stream API, 로깅, Spring 내부 호출 같은 노이즈를 제거해야 핵심 비즈니스 흐름에 집중할 수 있습니다.

**💡 Tip 4: 다이어그램 내에서 클릭으로 코드 이동**

생성된 다이어그램에서 **메서드 이름을 클릭하면 해당 소스 코드로 바로 이동**합니다. 다이어그램을 보다가 "이 부분 자세히 보고 싶다" 싶으면 바로 클릭!

> 📸 *[스크린샷: 다이어그램에서 메서드 클릭 시 해당 코드로 이동하는 모습]*

## 2. Call Hierarchy (IntelliJ 내장 기능)

### 2.1 시퀀스 다이어그램과 다른 점

SequenceDiagram 플러그인이 **"이 메서드 안에서 무슨 일이 벌어지나"** (Top-Down)에 답한다면, Call Hierarchy는 **"이 메서드에 누가 영향을 받나"** (Bottom-Up + Top-Down)에 답합니다.

| 관점 | SequenceDiagram | Call Hierarchy |
|------|----------------|----------------|
| 방향 | 호출하는 쪽 → 호출받는 쪽 (하향식) | 양방향 탐색 가능 |
| 형태 | 시퀀스 다이어그램 (시각적) | 트리 구조 (텍스트) |
| 강점 | 전체 흐름의 시각적 이해 | 영향 범위의 정확한 추적 |
| 주 용도 | "이 메서드가 뭘 하는지 파악" | "이 메서드를 고치면 어디가 깨지는지 파악" |

리팩토링에서는 두 도구가 **상호 보완적**입니다. 먼저 SequenceDiagram으로 전체 흐름을 파악한 뒤, 수정하고 싶은 메서드를 Call Hierarchy로 영향 범위를 확인하는 식이죠.

### 2.2 기본 사용법

**Step 1.** 분석하고 싶은 **메서드 이름 위에 커서**를 놓습니다.

**Step 2.** 단축키를 누릅니다:
- macOS: `⌃⌥H` (Control + Option + H)
- Windows/Linux: `Ctrl + Alt + H`

또는 메뉴에서 `Navigate → Call Hierarchy`를 선택합니다.

> 📸 *[스크린샷: processPayment() 메서드 위에서 Call Hierarchy 실행]*

**Step 3.** 하단에 Call Hierarchy 패널이 열립니다.

### 2.3 Caller Hierarchy — "이 메서드를 누가 호출하나?"

리팩토링에서 **가장 자주 쓰는 방향**입니다. 메서드의 시그니처를 변경하거나, 동작을 수정할 때 **영향 범위(blast radius)**를 파악하는 데 사용합니다.

예를 들어, `PaymentService.processPayment()`의 Caller Hierarchy를 확인해 보겠습니다.

> 📸 *[스크린샷: processPayment()의 Caller Hierarchy 트리]*

```
processPayment(PaymentRequest) — PaymentService
├── placeOrder(OrderCommand) — OrderFacade
│   ├── createOrder(CreateOrderRequest) — OrderController
│   │   └── (HTTP endpoint: POST /api/orders)
│   └── retryFailedOrder(String) — OrderRetryScheduler
│       └── (Scheduled task: @Scheduled cron)
├── processRefund(RefundRequest) — RefundService
│   └── handleRefund(RefundCommand) — RefundFacade
│       └── refund(RefundHttpRequest) — RefundController
│           └── (HTTP endpoint: POST /api/refunds)
└── reprocessPayment(String) — PaymentRetryService
    └── retryPendingPayments() — PaymentRetryScheduler
        └── (Scheduled task: @Scheduled cron)
```

이 트리 하나로 `processPayment()`의 **파라미터를 바꾸거나, 예외 처리 방식을 수정하면 영향받는 곳**이 한눈에 보입니다:

- `OrderFacade` (주문 처리)
- `RefundService` (환불 처리)  
- `PaymentRetryService` (결제 재시도)

3곳에서 호출하고 있으니, 시그니처 변경 시 3곳을 모두 수정해야 합니다.

### 2.4 Callee Hierarchy — "이 메서드가 뭘 호출하나?"

이번에는 방향을 바꿔서, `placeOrder()`가 **내부에서 호출하는 메서드들**을 트리로 확인합니다. Call Hierarchy 패널 상단의 방향 전환 버튼(🔄)을 클릭하거나 단축키로 전환합니다.

> 📸 *[스크린샷: Call Hierarchy 패널 상단의 Caller/Callee 전환 버튼]*

```
placeOrder(OrderCommand) — OrderFacade
├── reserve(List<ReservationRequest>) — InventoryService
│   ├── findByProductIdWithLock(String) — InventoryRepository
│   ├── lock(int) — Inventory
│   ├── save(InventoryReservation) — ReservationRepository
│   └── publish(InventoryReservedEvent) — InventoryEventPublisher
├── createOrder(OrderCommand, InventoryReservation) — OrderService
│   ├── validate(OrderCommand) — OrderValidator
│   ├── save(Order) — OrderRepository
│   └── mapToOrderItems(List<ItemCommand>) — OrderItemMapper
├── processPayment(PaymentRequest) — PaymentService
│   ├── validate(PaymentRequest) — PaymentValidator
│   ├── route(PaymentMethod) — PaymentGatewayRouter
│   ├── charge(Money, ChargeDetails) — PaymentGateway
│   └── save(Payment) — PaymentRepository
├── confirmOrder(String, PaymentResult) — OrderService
├── sendOrderConfirmation(Order) — NotificationService
│   ├── resolve(String, Order) — NotificationTemplateEngine
│   ├── resolveChannels(Customer, NotificationType) — NotificationChannelResolver
│   └── send(NotificationMessage) — NotificationChannel
├── publish(OrderPlacedEvent) — OrderEventPublisher
├── releaseReservation(String) — InventoryService  [catch 블록]
└── markOrderFailed(String, String) — OrderService  [catch 블록]
```

시퀀스 다이어그램과 비슷한 정보를 보여주지만, **트리 형태**라서 **특정 브랜치만 펼쳐보거나 접을 수 있는 장점**이 있습니다. 전체 흐름보다 **특정 경로를 깊이 추적**할 때 유리합니다.

### 2.5 실전: "PaymentValidator.validate()를 리팩토링하고 싶다면?"

구체적인 리팩토링 시나리오를 통해 Call Hierarchy의 진짜 가치를 체감해 봅시다.

**상황**: `PaymentValidator.validate()`의 검증 로직이 점점 복잡해져서, 파라미터를 `PaymentRequest` 대신 별도의 `ValidationContext`를 받도록 바꾸고 싶습니다.

```java
// AS-IS
public class PaymentValidator {
    public void validate(PaymentRequest request) {
        validateAmount(request.getAmount());
        validatePaymentMethod(request.getPaymentMethod());
        validateDuplicatePayment(request.getOrderId());
    }
}

// TO-BE
public class PaymentValidator {
    public void validate(ValidationContext context) {
        validateAmount(context.getAmount());
        validatePaymentMethod(context.getMethod());
        validateDuplicatePayment(context.getOrderId());
        validateCustomerCreditLimit(context.getCustomerId(), context.getAmount()); // 신규 검증 추가
    }
}
```

이때 Call Hierarchy로 **Caller를 확인**합니다:

```
validate(PaymentRequest) — PaymentValidator
├── processPayment(PaymentRequest) — PaymentService       ← 호출부 수정 필요
├── validateRefund(RefundRequest) — RefundValidator         ← 호출부 수정 필요
└── preCheckPayment(PaymentRequest) — PaymentCheckService  ← 호출부 수정 필요
```

3곳의 호출부를 모두 확인했으니, 이제 각 호출부에서 `ValidationContext`를 어떻게 생성해서 넘겨줄지 계획을 세울 수 있습니다. **하나라도 빠트리면 컴파일 에러는 잡아주겠지만**, 런타임에서야 드러나는 미묘한 동작 차이(예: 새로 추가된 검증 로직의 영향)까지 미리 파악하려면 이 트리가 필수입니다.

### 2.6 활용 팁

**💡 Tip 1: Scope 설정으로 노이즈 줄이기**

Call Hierarchy 패널 상단에서 **Scope**을 `Project Files`로 설정하면 테스트 코드나 라이브러리 호출은 제외됩니다.

> 📸 *[스크린샷: Call Hierarchy 패널의 Scope 드롭다운]*

반대로, 테스트 코드에서의 호출도 보고 싶다면 `All Places`로 설정합니다. 리팩토링 후 수정해야 할 테스트도 미리 파악할 수 있어서 유용합니다.

**💡 Tip 2: 인터페이스 메서드에서 실행하기**

인터페이스의 메서드에서 Call Hierarchy를 실행하면, **모든 구현체의 호출 관계**까지 추적됩니다. 예를 들어 `PaymentGateway.charge()`에서 실행하면 `TossPaymentGateway`, `KakaoPayGateway` 등 모든 구현체의 호출 관계가 트리에 나타납니다.

**💡 Tip 3: 리팩토링 전 "영향 범위 체크리스트" 만들기**

Call Hierarchy 결과를 보면서, 수정이 필요한 호출부를 TODO 코멘트로 미리 마킹해 두면 리팩토링 작업이 훨씬 안전해집니다.

```java
// TODO: [리팩토링] validate() 시그니처 변경에 따른 호출부 수정 필요
PaymentResult result = paymentService.processPayment(request);
```

## 3. 두 도구를 결합한 리팩토링 워크플로우

두 도구를 실제 리팩토링에 적용하는 워크플로우를 정리해 보겠습니다.

### Step 1: SequenceDiagram으로 전체 흐름 파악

리팩토링 대상 메서드(예: `placeOrder()`)에서 시퀀스 다이어그램을 생성합니다. 전체 흐름을 한 그림으로 보면서 **"어디가 복잡한가?"**, **"어디를 분리해야 하는가?"**를 판단합니다.

```
📊 시퀀스 다이어그램 관찰 결과:
 → OrderFacade에서 6개 객체로 직접 호출 → 오케스트레이션 과다
 → PaymentService 내부 4단계 호출 → 내부 복잡도 높음
 → 예외 처리 시 보상 트랜잭션 존재 → 분리 후보
```

### Step 2: 리팩토링 대상 선정

시퀀스 다이어그램에서 발견한 문제 중 가장 임팩트가 큰 것을 선택합니다. 예를 들어: **"결제 처리와 주문 확정을 하나의 단위로 묶자"**

### Step 3: Call Hierarchy로 영향 범위 확인

선정한 리팩토링 대상 메서드(`processPayment`, `confirmOrder`)의 Caller Hierarchy를 확인합니다. 영향받는 모든 호출부를 리스트업합니다.

```
📋 영향 범위 체크리스트:
 □ OrderFacade.placeOrder() — 주문 처리
 □ OrderRetryScheduler.retryFailedOrder() — 재시도 처리
 □ RefundService.processRefund() — 환불 처리
 □ PaymentRetryService.reprocessPayment() — 결제 재시도
 □ OrderFacadeTest — 단위 테스트
 □ PaymentServiceIntegrationTest — 통합 테스트
```

### Step 4: 리팩토링 수행

영향 범위를 명확히 파악한 상태에서 안전하게 코드를 수정합니다.

### Step 5: SequenceDiagram으로 결과 검증

리팩토링 후 다시 시퀀스 다이어그램을 생성해서, **의도한 대로 흐름이 변경되었는지** 확인합니다.

> 이 워크플로우는 1부에서 다뤘던 PlantUML의 AS-IS / TO-BE 비교와도 연결됩니다. 정적 구조(1부)와 동적 흐름(2부)을 함께 비교하면, 리팩토링의 효과를 더 입체적으로 검증할 수 있습니다.

## 4. 주의 사항과 한계

### SequenceDiagram 플러그인 (Free 버전)

| 항목 | 내용 |
|------|------|
| **호출 깊이 제한** | Free 버전에서는 설정할 수 있는 call depth에 제한이 있을 수 있습니다. 복잡한 체인을 끝까지 추적하려면 중간 메서드에서 별도로 다이어그램을 생성하세요. |
| **동적 디스패치** | 인터페이스나 추상 클래스를 통한 호출은 실제 구현체까지 추적되지 않을 수 있습니다. 이 경우 어떤 구현체가 실행되는지는 코드를 직접 확인해야 합니다. |
| **비동기 호출** | `@Async`, `CompletableFuture`, 이벤트 기반 호출 등은 다이어그램에 표현되지 않습니다. |
| **내보내기 제한** | Free 버전에서는 PlantUML/Mermaid 소스 내보내기가 제한됩니다. 스크린샷으로 캡처하는 방식으로 대체하세요. |

### Call Hierarchy

| 항목 | 내용 |
|------|------|
| **리플렉션 호출** | `Method.invoke()` 등 리플렉션으로 호출되는 경우 추적되지 않습니다. |
| **Spring 프록시** | AOP 프록시를 통한 호출은 트리에 나타나지 않을 수 있습니다. |
| **동적 디스패치** | 런타임 타입에 따른 메서드 해석은 정적 분석의 한계입니다. |
| **대규모 프로젝트** | 프로젝트가 크면 Caller 트리가 매우 깊어질 수 있습니다. Scope 설정을 잘 활용하세요. |

두 도구 모두 **정적 분석** 기반이라는 공통된 한계가 있습니다. 런타임에 결정되는 동적 바인딩, 리플렉션, 이벤트 리스너 체인 등은 도구만으로 완벽하게 추적하기 어렵습니다. 중요한 흐름은 **디버거**나 **런타임 로그 추적**을 병행하는 것이 좋습니다.

## 마치며

2부에서는 **동적인 메서드 호출 흐름**을 파악하는 두 가지 도구를 살펴봤습니다.

- **SequenceDiagram 플러그인**: 메서드 내부의 호출 순서를 시각적으로 확인 → *"이 메서드가 무슨 일을 하는지"* 파악
- **Call Hierarchy**: 메서드의 호출 관계를 트리로 추적 → *"이 메서드를 고치면 어디까지 영향이 가는지"* 파악

1부의 **정적 구조 분석**(클래스 의존성 다이어그램)과 2부의 **동적 흐름 분석**(시퀀스 다이어그램 + Call Hierarchy)을 함께 활용하면, 리팩토링 전에 코드를 **구조적으로도, 행동적으로도** 이해한 상태에서 안전하게 손댈 수 있습니다.

3부에서는 한 단계 더 올라가, **아키텍처/모듈 단위**의 의존성 문제를 분석합니다. IntelliJ의 DSM(Dependency Structure Matrix)을 활용해 패키지 간 순환 참조와 얽힌 의존성을 찾아내는 방법을 다룰 예정이니 기대해 주세요!
