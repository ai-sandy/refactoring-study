# Refactoring을 도와주는 도구들 3부 — IntelliJ DSM으로 패키지 간 순환 참조와 얽힌 의존성 해부하기

> 1부에서는 IntelliJ + PUML로 **정적인 클래스 의존성**을 봤고, 2부에서는 Sequence Diagram 플러그인과 Call Hierarchy로 **동적인 로직 흐름**을 따라갔다. 3부에서는 한 단계 더 올라가서, **패키지/모듈 단위의 구조적 얽힘**을 다룬다.
>
> 클래스 다이어그램과 시퀀스 다이어그램만으로는 “어느 패키지부터 끊어야 하는가?”가 잘 보이지 않을 때가 있다. 이럴 때 IntelliJ의 DSM(Dependency Structure Matrix)은 리팩터링의 출발점을 꽤 선명하게 보여준다.

이 글의 목표는 하나다. **패키지 간 순환 참조와 과도한 얽힘을 눈으로 확인하고, 그 결과를 바탕으로 어디서부터 의존성을 끊을지 결정하는 방법**을 실제 코드 예시와 함께 정리하는 것이다.

## 왜 3부에서 DSM인가

리팩터링을 하다 보면 보통 이런 순서로 막힌다.

- 클래스 레벨에서는 문제를 이미 봤다.
- 메서드 흐름도 따라가 봤다.
- 그런데 정작 패키지를 나누거나 모듈을 분리하려고 하면, 어디가 중심이고 어디가 주변인지 감이 안 온다.

특히 아래 같은 상황에서 그렇다.

- `order`가 `payment`, `inventory`, `shipping`을 호출하는 것은 자연스러워 보인다.
- 그런데 어느 순간 `payment`도 다시 `order`를 참조하고,
- `shipping`은 `inventory`를 보고,
- `inventory`는 다시 `order`를 받기 시작한다.

이 상태가 되면 개별 클래스만 봐서는 “왜 이렇게 고치기 어려운지”를 설명하기 어렵다. 하지만 DSM으로 보면 **어떤 패키지 쌍이 서로 물고 있는지**, **어느 방향으로 의존성이 흘러야 하는지**, **빨간 셀이 어디에 몰려 있는지**가 한 번에 드러난다.

## 준비물

- IntelliJ IDEA Ultimate
- `Dependency Structure Matrix` 플러그인 활성화
- 빌드 가능한 프로젝트 상태

여기서 한 가지는 미리 적고 가야 한다. 이번 글은 **서드파티 유료 플러그인 기능은 사용하지 않는다.** 다만 DSM 자체는 현재 기준으로 **IntelliJ IDEA Ultimate 환경에서만 사용 가능한 JetBrains 공식 플러그인**이다. 즉, “추가 유료 플러그인을 사서 쓰는 글”은 아니지만, Community만으로 그대로 따라 하기는 어렵다.

## 이번 글에서 볼 예제

예제는 전형적인 커머스 주문 흐름이다. 처음에는 “기능별 패키지 분리”가 그럴듯해 보이지만, 시간이 지나면서 서로의 내부 모델을 가져다 쓰기 시작한 상태를 가정한다.

```text
com.example.commerce
├─ order
│  ├─ application
│  │  ├─ PlaceOrderService
│  │  └─ OrderQueryService
│  └─ domain
│     ├─ Order
│     └─ OrderLine
├─ payment
│  ├─ application
│  │  └─ PaymentService
│  └─ domain
│     └─ LoyaltyDiscountPolicy
├─ inventory
│  └─ application
│     └─ InventoryAllocator
├─ shipping
│  └─ application
│     └─ ShippingFeeService
└─ promotion
   └─ application
      └─ CouponGrantService
```

겉으로 보면 나쁘지 않다. 그런데 실제 코드를 보면 의존성이 양방향으로 섞이기 시작한다.

## 문제 코드: 패키지는 나뉘어 있지만 의존성 방향은 무너진 상태

아래 예시는 설명을 위해 일부 DTO/VO를 생략했지만, 패턴 자체는 현업 코드에서 자주 보는 형태다.

### 1) `order`가 여러 패키지를 오케스트레이션한다

```java
package com.example.commerce.order.application;

@Service
@RequiredArgsConstructor
@Transactional
public class PlaceOrderService {
    private final OrderRepository orderRepository;
    private final ShippingFeeService shippingFeeService;
    private final InventoryAllocator inventoryAllocator;
    private final PaymentService paymentService;
    private final CouponGrantService couponGrantService;

    public PlaceOrderResult place(PlaceOrderCommand command) {
        Order order = Order.create(
                command.customerId(),
                command.lines(),
                command.shippingAddress(),
                command.requestedAt(),
                command.couponCode());

        Money shippingFee = shippingFeeService.calculate(order);
        AllocationResult allocation = inventoryAllocator.allocate(order);

        PaymentReceipt receipt = paymentService.authorize(
                order,
                shippingFee,
                allocation.reservationExpiresAt());

        order.confirm(
                receipt.transactionId(),
                shippingFee,
                allocation.reservedItems());

        orderRepository.save(order);

        if (order.payableAmount().isGreaterThan(Money.wons(200_000))) {
            couponGrantService.grantPostPurchaseCoupon(order.getId());
        }

        return PlaceOrderResult.of(order, allocation, receipt);
    }
}
```

여기까지는 얼핏 자연스럽다. 문제는 **다른 패키지들이 다시 `order` 쪽 내부 모델과 서비스를 끌어다 쓰기 시작할 때** 생긴다.

### 2) `payment`가 다시 `order`의 애플리케이션 서비스와 도메인 모델을 참조한다

```java
package com.example.commerce.payment.application;

@Service
@RequiredArgsConstructor
public class PaymentService {
    private final PaymentGateway paymentGateway;
    private final OrderQueryService orderQueryService;
    private final LoyaltyDiscountPolicy loyaltyDiscountPolicy;

    public PaymentReceipt authorize(
            Order order,
            Money shippingFee,
            LocalDateTime reservationExpiresAt) {

        OrderSummary summary = orderQueryService.load(order.getId());

        Money loyaltyDiscount = loyaltyDiscountPolicy.discount(
                summary.customerGrade(),
                summary.firstOrder(),
                reservationExpiresAt);

        Money chargeAmount = order.productsTotal()
                .plus(shippingFee)
                .minus(order.appliedCouponDiscount())
                .minus(loyaltyDiscount);

        PaymentRequest request = new PaymentRequest(
                order.getId().value(),
                summary.customerId().value(),
                chargeAmount,
                order.containsColdChainItem(),
                order.requestedAt());

        return paymentGateway.authorize(request);
    }
}
```

여기서 이미 `order → payment`는 물론이고, `payment → order`도 생겼다.

- `PaymentService`가 `Order` aggregate를 파라미터로 받는다.
- `OrderQueryService`까지 직접 호출한다.

즉, `payment`는 더 이상 독립된 정책 패키지가 아니라, `order` 내부 구조를 알아야만 동작하는 패키지가 된다.

### 3) `inventory`는 `order` 모델을 직접 받아 재고 예약을 처리한다

```java
package com.example.commerce.inventory.application;

@Service
@RequiredArgsConstructor
public class InventoryAllocator {
    private final InventoryRepository inventoryRepository;
    private final BackOrderPolicy backOrderPolicy;

    public AllocationResult allocate(Order order) {
        List<ReservedItem> reserved = new ArrayList<>();

        for (OrderLine line : order.lines()) {
            Stock stock = inventoryRepository.get(line.productId());
            int reservedQty = stock.reserve(
                    line.quantity(),
                    ReservationPriority.from(order.requestedAt()),
                    backOrderPolicy.allowedDays(order.customerId(), line.productId()));

            reserved.add(new ReservedItem(line.productId(), reservedQty));
        }

        Duration ttl = order.containsColdChainItem()
                ? Duration.ofMinutes(20)
                : Duration.ofHours(2);

        return AllocationResult.of(reserved, ttl);
    }
}
```

`inventory`도 `Order`, `OrderLine`에 직접 묶였다. 처음에는 편하지만, 나중에는 `order` 도메인 변경이 `inventory`까지 연쇄적으로 흔든다.

### 4) `shipping`은 다시 `order`와 `inventory`를 같이 본다

```java
package com.example.commerce.shipping.application;

@Service
@RequiredArgsConstructor
public class ShippingFeeService {
    private final InventoryReadService inventoryReadService;
    private final CarrierRateTable carrierRateTable;

    public Money calculate(Order order) {
        ShippingZone zone = ShippingZone.from(order.shippingAddress());
        boolean bulky = order.totalWeightKg() > 20.0
                || order.lines().stream().anyMatch(OrderLine::isOversized);

        long nearStockoutItems = order.lines().stream()
                .filter(line -> inventoryReadService.availableQuantity(line.productId()) < line.quantity() * 2)
                .count();

        Money baseRate = carrierRateTable.baseRate(
                zone,
                bulky,
                order.requestedAt(),
                order.containsColdChainItem());

        Money scarcitySurcharge = Money.wons((int) nearStockoutItems * 1_000);
        Money vipDiscount = order.isMemberVip() ? Money.wons(2_500) : Money.ZERO;

        return baseRate.plus(scarcitySurcharge).minus(vipDiscount);
    }
}
```

이제 `shipping → order`, `shipping → inventory`가 생긴다.

### 5) `promotion`도 후속 쿠폰 발급을 위해 `order` 조회 서비스에 붙는다

```java
package com.example.commerce.promotion.application;

@Service
@RequiredArgsConstructor
public class CouponGrantService {
    private final OrderQueryService orderQueryService;
    private final CouponRepository couponRepository;

    public void grantPostPurchaseCoupon(OrderId orderId) {
        OrderSummary order = orderQueryService.load(orderId);

        if (order.totalAmount().isGreaterThan(Money.wons(200_000))
                && !order.firstOrder()) {
            couponRepository.save(
                    Coupon.repurchaseBonus(
                            order.customerId(),
                            LocalDate.now().plusDays(30)));
        }
    }
}
```

정리하면 패키지 레벨에서는 대략 이런 그림이 된다.

```text
order      -> payment, inventory, shipping, promotion
payment    -> order
inventory  -> order
shipping   -> order, inventory
promotion  -> order
```

클래스 단위로는 각각 “그럴싸한 이유”가 있어 보인다. 하지만 패키지 단위에서는 이미 다음 문제가 시작됐다.

- `order ↔ payment` 순환 참조
- `order ↔ inventory` 순환 참조
- `order ↔ shipping` 순환 참조
- `shipping → inventory` 추가 결합
- `promotion → order`로 인해 후처리 로직도 `order` 내부 조회 구조에 종속

이쯤 되면 어느 클래스 하나만 정리해서는 해결되지 않는다. 바로 이 지점에서 DSM이 유용하다.

## IntelliJ DSM 실행하기

DSM은 “내가 느끼기에 복잡하다”를 **실제 구조로 시각화**해 준다.

1. `Code | Analyze Code | Dependency Matrix`를 연다.
2. 전체 프로젝트보다 먼저 **문제가 의심되는 production scope**를 고른다.
3. 가능하면 테스트 코드는 처음 분석에서는 제외한다.
4. 빌드가 최신 상태가 아니면 먼저 컴파일한다.

실무에서는 보통 처음부터 전체 저장소를 보는 것보다, 아래처럼 범위를 줄이는 편이 낫다.

```text
src/main/java/com/example/commerce/order..
src/main/java/com/example/commerce/payment..
src/main/java/com/example/commerce/inventory..
src/main/java/com/example/commerce/shipping..
src/main/java/com/example/commerce/promotion..
```

테스트 코드를 같이 넣으면 fixture, test helper, mock configuration 때문에 순환 참조가 과장돼 보이는 경우가 많다. 첫 번째 진단은 **production 코드만** 보는 편이 좋다.

## DSM은 어떻게 읽어야 하나

DSM을 처음 보면 조금 낯설다. 그래프가 아니라 표처럼 보이기 때문이다. 하지만 몇 가지만 기억하면 금방 읽힌다.

### 1) 처음에는 모듈 기준으로 보이고, 펼치면 패키지까지 내려간다

처음에는 모듈만 보일 수 있다. 여기서 트리를 펼치면 패키지, 더 들어가면 클래스 수준까지 내려가며 볼 수 있다.

### 2) 빨간 셀은 서로 물고 있다는 뜻이다

서로 양방향 의존이 있거나 순환이 있으면 빨간색으로 보인다. 패키지 리팩터링에서 가장 먼저 볼 것은 이 빨간 셀이다.

### 3) 색이 진할수록 참조 수가 많다

한두 군데가 아니라 여러 클래스가 서로를 많이 참조하고 있다면 색이 더 짙어진다. 즉, 단순한 우발적 의존인지, 구조적으로 얽혀 있는지 감이 온다.

### 4) 기억할 규칙은 하나면 충분하다

**의존성은 Green 에서 Yellow 방향으로 흐른다.**

이 규칙만 기억하면 “누가 누구에게 기대고 있는지”를 읽는 데 큰 도움이 된다. 빨간 셀은 그 방향성이 무너졌다는 신호다.

## 예제 프로젝트를 DSM으로 보면 무엇이 보이나

위 예제를 DSM으로 열고 `order`, `payment`, `inventory`, `shipping`, `promotion` 패키지를 펼치면 보통 다음 순서로 읽게 된다.

### 먼저 `order` 주변의 빨간 셀을 찾는다

가장 먼저 `order` 행과 열 근처를 본다. 이 패키지는 대개 중심 오케스트레이션 패키지라서 의존성이 많이 모인다. 문제는 “많이 참조한다”가 아니라 “다시 참조당한다”는 것이다.

예상되는 첫 번째 후보는 다음 둘이다.

- `order ↔ payment`
- `order ↔ inventory`

여기서 중요한 것은 “결제와 재고가 주문을 참조하면 안 된다” 같은 교과서 문장을 외우는 게 아니다. **왜 다시 참조하게 됐는지**를 코드로 연결해서 보는 것이다.

우리 예제에서는 원인이 비교적 선명하다.

- `payment`는 `Order` aggregate와 `OrderQueryService`를 직접 사용한다.
- `inventory`는 `Order` 전체를 그대로 받아 사용한다.
- `shipping`은 `Order`와 `InventoryReadService`를 동시에 본다.
- `promotion`은 후처리를 위해 `OrderQueryService`에 붙어 있다.

DSM에서 빨간 셀을 클릭해 보면 “아, 실제로는 `payment` 패키지의 몇몇 클래스가 `order.application`, `order.domain`을 동시에 물고 있구나”가 숫자와 함께 나온다. 그때부터는 추상적인 논쟁이 아니라, **어떤 참조를 끊어야 하는지**가 작업 단위로 바뀐다.

## DSM에서 바로 다음으로 해야 할 일

빨간 셀을 찾았다고 바로 리팩터링에 들어가면 다시 막힌다. 그 다음 순서가 중요하다.

### 1) 셀을 클릭해서 “숫자”를 먼저 본다

순환이 있다는 사실보다 더 중요한 것은 **얼마나 많이** 엮여 있는가다.

- 참조 수가 1~2개면 국소 수정으로 끊을 수 있다.
- 참조 수가 수십 개면 패키지 책임 자체를 다시 나눠야 할 가능성이 높다.

### 2) `Explore Dependencies Between`으로 실제 원인 클래스만 좁힌다

DSM에서 패키지 전체를 보고 있으면 여전히 범위가 넓다. 이때 셀에서 우클릭해서 **`Explore Dependencies Between`**를 쓰면, 그 의존성을 만들어내는 클래스들만 별도 탭으로 좁혀서 볼 수 있다.

이 기능이 특히 좋은 이유는, “`payment`가 `order`에 의존한다”라는 말이 너무 큰 덩어리일 때 **실제로는 `PaymentService`, `LoyaltyDiscountPolicy`, `OrderQueryService` 연결이 핵심인지**를 바로 확인할 수 있기 때문이다.

### 3) `Find Usages for Dependencies`로 소스까지 연결한다

패키지 레벨에서 원인 클래스를 좁혔다면, 이제는 해당 참조가 실제로 어떤 import와 어떤 메서드 호출로 생겼는지 소스까지 내려가야 한다. 이때 **`Find Usages for Dependencies`**가 유용하다.

즉, DSM은 “문제를 찾는 도구”에서 끝나는 게 아니라, **실제 수정할 import 라인까지 내려가는 출발점**으로 써야 한다.

## 이 예제에서 어디를 먼저 끊어야 하나

예제 기준으로는 `payment → order`와 `inventory → order`를 먼저 끊는 것이 가장 효과적이다.

왜냐하면 이 둘은 공통적으로 같은 냄새를 갖고 있기 때문이다.

- 패키지 경계를 넘어 **`Order` aggregate 전체를 통째로 넘긴다.**
- 자신의 계산에 필요한 데이터만 받지 않고, 다른 패키지의 내부 모델에 직접 올라탄다.
- 그 결과 `order` 구조가 바뀌면 `payment`, `inventory`, `shipping`까지 같이 흔들린다.

리팩터링 관점에서 보면 이 문제의 핵심은 대개 아래 둘 중 하나다.

1. **오케스트레이션 책임이 섞였다.**
   - 원래는 상위 애플리케이션 서비스가 조합해야 할 데이터를 하위 패키지가 직접 끌어오고 있다.
2. **경계 간 계약이 없다.**
   - `PaymentRequest`, `ReservationCommand`, `ShippingQuoteQuery`처럼 경계용 DTO/명령 객체가 없어서, 가장 가까운 내부 모델인 `Order`를 그냥 넘겨버린다.

## 리팩터링 전략: Aggregate를 넘기지 말고 경계 계약을 만든다

이 예제에서는 다음 원칙으로 풀 수 있다.

### 원칙 1) `Order` 전체를 넘기지 않는다

`payment`, `inventory`, `shipping`이 필요로 하는 것은 `Order` 전체가 아니다.

- 결제는 청구 금액, 고객 ID, 주문 시각, 냉장 배송 여부 정도면 된다.
- 재고는 상품 ID, 수량, 고객 ID, 예약 우선순위 정도면 된다.
- 배송은 주소, 품목 특성, 고객 등급 정도면 된다.

즉, 각 패키지에 **자기 계산에 필요한 데이터만 담은 계약 객체**를 넘기면 된다.

### 원칙 2) 상위 오케스트레이션 계층을 분리한다

주문 생성, 배송비 산정, 재고 예약, 결제 승인, 후속 이벤트 발행은 사실상 하나의 유스케이스다. 이 조합 책임을 `order` 패키지 안에 계속 넣어두면 다른 패키지가 다시 `order`에 끌려 들어오기 쉽다.

그래서 이번 예제에서는 `checkout` 패키지를 새로 만들고, 유스케이스 오케스트레이션을 그쪽으로 올리겠다.

### 원칙 3) 후처리는 직접 호출보다 이벤트로 늦춘다

`promotion`이 쿠폰 발급을 위해 `OrderQueryService`를 직접 조회하는 대신, 주문 완료 이벤트를 구독하게 만들면 `promotion → order` 의존성을 줄일 수 있다.

## 리팩터링 후 코드

아래는 같은 시나리오를 패키지 경계를 살려 다시 정리한 예시다.

### 1) 오케스트레이션은 `checkout`이 담당한다

```java
package com.example.commerce.checkout.application;

@Service
@RequiredArgsConstructor
@Transactional
public class CheckoutFacade {
    private final OrderRepository orderRepository;
    private final ShippingFeeService shippingFeeService;
    private final InventoryAllocator inventoryAllocator;
    private final PaymentService paymentService;
    private final ApplicationEventPublisher eventPublisher;

    public CheckoutResult place(PlaceOrderCommand command) {
        Order order = Order.create(
                command.customerId(),
                command.lines(),
                command.shippingAddress(),
                command.requestedAt(),
                command.couponCode());

        ShippingQuote shippingQuote = shippingFeeService.quote(
                new ShippingQuoteQuery(
                        command.shippingAddress(),
                        OrderLineSnapshot.from(command.lines()),
                        command.requestedAt(),
                        command.customerGrade(),
                        command.coldChain()));

        AllocationResult allocation = inventoryAllocator.allocate(
                new ReservationCommand(
                        order.getId().value(),
                        command.customerId().value(),
                        OrderLineSnapshot.from(command.lines()),
                        shippingQuote.expressAvailable(),
                        command.requestedAt(),
                        command.coldChain()));

        PaymentReceipt receipt = paymentService.authorize(
                new PaymentCommand(
                        order.getId().value(),
                        command.customerId().value(),
                        order.productsTotal(),
                        order.appliedCouponDiscount(),
                        shippingQuote.fee(),
                        command.customerGrade(),
                        command.firstOrder(),
                        allocation.reservationExpiresAt(),
                        command.coldChain(),
                        command.requestedAt()));

        order.confirm(
                receipt.transactionId(),
                shippingQuote.fee(),
                allocation.reservedItems());

        orderRepository.save(order);

        eventPublisher.publishEvent(
                new OrderCompletedEvent(
                        order.getId().value(),
                        command.customerId().value(),
                        order.payableAmount()));

        return CheckoutResult.of(order, allocation, receipt, shippingQuote);
    }
}
```

핵심은 `checkout`이 조합하고, 다른 패키지는 **자기 일에 필요한 데이터만** 받도록 만든 점이다.

### 2) `payment`는 이제 `order`를 모른다

```java
package com.example.commerce.payment.application;

@Service
@RequiredArgsConstructor
public class PaymentService {
    private final PaymentGateway paymentGateway;
    private final LoyaltyDiscountPolicy loyaltyDiscountPolicy;

    public PaymentReceipt authorize(PaymentCommand command) {
        Money loyaltyDiscount = loyaltyDiscountPolicy.discount(
                command.customerGrade(),
                command.firstOrder(),
                command.reservationExpiresAt());

        Money chargeAmount = command.productsTotal()
                .plus(command.shippingFee())
                .minus(command.couponDiscount())
                .minus(loyaltyDiscount);

        PaymentRequest request = new PaymentRequest(
                command.orderId(),
                command.customerId(),
                chargeAmount,
                command.coldChain(),
                command.requestedAt());

        return paymentGateway.authorize(request);
    }
}
```

이제 `payment`는 더 이상 `Order` aggregate나 `OrderQueryService`를 몰라도 된다. `payment → order` 간선이 사라진다.

### 3) `inventory`도 `Order` 대신 경계 계약만 받는다

```java
package com.example.commerce.inventory.application;

@Service
@RequiredArgsConstructor
public class InventoryAllocator {
    private final InventoryRepository inventoryRepository;
    private final BackOrderPolicy backOrderPolicy;

    public AllocationResult allocate(ReservationCommand command) {
        ReservationPriority priority = command.expressAvailable()
                ? ReservationPriority.EXPRESS
                : ReservationPriority.STANDARD;

        List<ReservedItem> reserved = new ArrayList<>();

        for (OrderLineSnapshot line : command.lines()) {
            Stock stock = inventoryRepository.get(line.productId());
            int reservedQty = stock.reserve(
                    line.quantity(),
                    priority,
                    backOrderPolicy.allowedDays(command.customerId(), line.productId()));

            reserved.add(new ReservedItem(line.productId(), reservedQty));
        }

        Duration ttl = command.coldChain()
                ? Duration.ofMinutes(20)
                : Duration.ofHours(2);

        return AllocationResult.of(reserved, ttl);
    }
}
```

이제 `inventory`는 `order.domain.Order`와 `OrderLine`에 묶이지 않는다. `inventory → order`도 같이 끊긴다.

### 4) `shipping`은 조회 계약만 받고 견적만 계산한다

```java
package com.example.commerce.shipping.application;

@Service
@RequiredArgsConstructor
public class ShippingFeeService {
    private final InventoryReadService inventoryReadService;
    private final CarrierRateTable carrierRateTable;

    public ShippingQuote quote(ShippingQuoteQuery query) {
        ShippingZone zone = ShippingZone.from(query.shippingAddress());
        boolean bulky = query.lines().stream()
                .anyMatch(line -> line.weightKg() > 20.0 || line.oversized());

        long nearStockoutItems = query.lines().stream()
                .filter(line -> inventoryReadService.availableQuantity(line.productId()) < line.quantity() * 2)
                .count();

        Money baseRate = carrierRateTable.baseRate(
                zone,
                bulky,
                query.requestedAt(),
                query.coldChain());

        Money scarcitySurcharge = Money.wons((int) nearStockoutItems * 1_000);
        Money vipDiscount = query.customerGrade().isVip()
                ? Money.wons(2_500)
                : Money.ZERO;

        return new ShippingQuote(
                baseRate.plus(scarcitySurcharge).minus(vipDiscount),
                zone,
                nearStockoutItems > 0);
    }
}
```

`shipping` 역시 `Order` aggregate에 직접 묶이지 않는다.

### 5) `promotion`은 주문 조회 대신 이벤트를 구독한다

```java
package com.example.commerce.promotion.application;

@Component
@RequiredArgsConstructor
public class CouponGrantHandler {
    private final CouponRepository couponRepository;

    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void handle(OrderCompletedEvent event) {
        if (event.payableAmount().isGreaterThan(Money.wons(200_000))) {
            couponRepository.save(
                    Coupon.repurchaseBonus(
                            new CustomerId(event.customerId()),
                            LocalDate.now().plusDays(30)));
        }
    }
}
```

이렇게 바꾸면 `promotion`은 `OrderQueryService`에 올라타지 않아도 된다.

## 리팩터링 후 DSM에서 기대하는 변화

리팩터링 후에는 대략 아래 같은 방향이 된다.

```text
checkout   -> order, payment, inventory, shipping, promotion
shipping   -> inventory
payment    -> (order에 대한 역참조 없음)
inventory  -> (order에 대한 역참조 없음)
promotion  -> (order 조회 서비스에 대한 직접 의존 없음)
```

이 상태에서 DSM을 다시 열어 보면 기대하는 변화는 명확하다.

- `order ↔ payment` 빨간 셀이 사라진다.
- `order ↔ inventory` 빨간 셀이 사라진다.
- `order ↔ shipping`도 직접 양방향 참조가 줄어든다.
- 빨간 셀의 개수가 줄고, 의존성 흐름이 더 한 방향으로 정리된다.

이때 중요한 것은 “빨간 셀이 0이 되었는가”만이 아니다. 더 중요한 것은 **의존성의 방향을 설명할 수 있게 되었는가**다.

예를 들면 이렇게 말할 수 있어야 한다.

- `checkout`은 유스케이스를 조합하는 상위 계층이다.
- `payment`, `inventory`, `shipping`은 자기 계약을 통해 협력한다.
- `promotion`은 후처리 정책으로서 이벤트를 구독한다.

DSM이 좋은 이유는 이 설명이 단순한 설계 문장이 아니라, **실제 코드 의존성과 일치하는지**를 바로 확인할 수 있다는 점이다.

## 현업에서 DSM을 볼 때 자주 쓰는 체크리스트

실제로는 한 번에 완벽하게 정리되지 않는다. 그래서 아래 순서로 보는 편이 효율적이었다.

### 체크리스트 1) 빨간 셀부터 본다

전체를 다 이해하려고 하면 오래 걸린다. 먼저 빨간 셀부터 본다.

질문은 단순하다.

- 이 양방향 참조는 정말 필요한가?
- 아니면 편해서 aggregate를 통째로 넘긴 결과인가?
- 오케스트레이션 책임이 하위 패키지로 새어 들어간 것은 아닌가?

### 체크리스트 2) 패키지 간 계약 객체가 있는지 본다

패키지 경계에 아래가 없으면 거의 높은 확률로 내부 모델이 새어 나온다.

- command/query DTO
- snapshot/read model
- event contract
- port/interface

즉, **경계 계약이 없으면 경계도 없다**고 보는 편이 낫다.

### 체크리스트 3) Aggregate를 넘기는 코드부터 의심한다

아래와 같은 시그니처가 보이면 거의 항상 DSM에서 역참조가 생긴다.

```java
paymentService.authorize(Order order, ...)
inventoryAllocator.allocate(Order order)
shippingFeeService.calculate(Order order)
```

메서드가 편하다고 구조도 좋은 것은 아니다. 다른 패키지에 `Order`를 넘기는 순간, 그 패키지는 언젠가 `order.domain`의 세부사항을 알게 된다.

### 체크리스트 4) 후처리는 직접 호출보다 이벤트를 먼저 검토한다

`couponGrantService.grantPostPurchaseCoupon(orderId)`처럼 직접 호출이 늘어나기 시작하면, 후처리 패키지가 핵심 패키지 내부 조회 구조에 묶이기 쉽다. 반드시 이벤트가 답이라는 뜻은 아니지만, **후처리가 핵심 트랜잭션의 내부 모델을 다시 조회하고 있다면 구조 신호를 한 번 의심할 만하다.**

### 체크리스트 5) 테스트는 처음 분석에서 빼 본다

테스트 편의 코드 때문에 production보다 더 복잡한 의존성이 보일 때가 많다. 첫 번째 DSM은 production only로, 두 번째 DSM은 test 포함으로 보는 식이 낫다.

## DSM이 특히 강한 순간

개인적으로 DSM은 아래 상황에서 가장 강했다.

### 1) 패키지 분리나 모듈 분리를 앞두고 있을 때

“이 패키지를 밖으로 빼도 되나?”를 감으로 결정하지 않고, **실제 역참조가 얼마나 남아 있는지** 확인할 수 있다.

### 2) 리팩터링 우선순위를 정해야 할 때

모든 냄새를 한 번에 고칠 수는 없다. DSM은 어느 패키지 쌍이 가장 많이 얽혀 있는지 보여주므로, **가성비 좋은 첫 번째 절단면**을 고르기 좋다.

### 3) 설계 리뷰를 추상 논쟁이 아니라 코드 근거로 하고 싶을 때

“이 구조가 별로예요”보다 “`payment ↔ order`가 지금 이렇게 물려 있고, 실제 참조 수가 이 정도라서 먼저 끊자”가 훨씬 생산적이다.

## 마무리

1부에서 본 클래스 다이어그램은 **정적인 구조를 넓게** 보는 데 좋았고, 2부의 시퀀스 다이어그램과 Call Hierarchy는 **동적인 호출 흐름을 깊게** 보는 데 좋았다. 3부의 DSM은 그 사이 어딘가, 정확히는 **리팩터링할 구조적 경계를 결정하는 순간**에 특히 강하다.

이번 글의 요지를 한 문장으로 줄이면 이렇다.

> **리팩터링은 코드부터 고치는 작업이 아니라, 끊어야 할 의존성을 먼저 찾는 작업이다.**

DSM은 바로 그 “어디를 끊어야 하는가”를 패키지/모듈 단위에서 보여준다. 클래스와 메서드 수준에서 이미 감지한 문제를, 아키텍처 수준에서 우선순위로 바꾸고 싶을 때 가장 먼저 열어볼 만한 화면이다.

### 시리즈 정리

- 1부: `IntelliJ + PUML`로 정적인 클래스 의존성 다이어그램 그리기
- 2부: `Sequence Diagram 플러그인 + Call Hierarchy`로 동적인 로직 흐름 파악하기
- 3부: `IntelliJ DSM`으로 패키지 간 순환 참조 및 얽힌 의존성 분석하기

필요하다면 다음 리비전에서는 이 글의 후속편으로, DSM에서 찾은 순환 참조를 실제로 끊을 때 자주 쓰는 리팩터링 패턴(Orchestrator 추출, Contract DTO 도입, Port 분리, Event 전환)만 따로 묶어서 정리해도 괜찮다.
